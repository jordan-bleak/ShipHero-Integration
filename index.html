The Problem
Warehouse rejections don't have a standard status in ShipHero
Every warehouse names its rejection statuses differently. "Damaged." "Not received." "Wrong item." "Fraud." ShipHero doesn't emit a universal "rejected" signal — it just sends whatever status the warehouse team entered.
Redo's old behavior: anything it didn't recognize landed in a manual review queue. Ops teams had to triage those returns by hand, status by status, over and over.
The result was a backlog of returns sitting in limbo — invisible to customer service, inconsistent in finance, and a daily burden for whoever owned the queue.
How the Integration Works
Redo and ShipHero stay in sync across the full return lifecycle
What's New
New Feature
Configurable rejection statuses
ShipHero doesn't emit a standard "rejected" event — every warehouse uses its own status names. Now you can tell Redo exactly which statuses your warehouse uses for rejections. When ShipHero sends one of them, Redo automatically marks the return (and every line item on it) as rejected and logs a timeline entry. No engineering work. No Redo-side mapping table to maintain. Your warehouse's vocabulary, respected as-is.
Also Included
Graceful handling of warehouse corrections
Warehouses make mistakes. An item flagged as damaged gets re-inspected and approved. ShipHero updates the status to "complete."
Redo catches that correction automatically. If a return that was marked rejected receives a "complete" status from ShipHero, Redo reopens it and processes it normally — no manual intervention needed. The warehouse's source-of-truth decision is reflected in Redo in real time.
